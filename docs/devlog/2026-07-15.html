画像も、ちゃんとサイトへ

今日はアプリ本体だけでなく、ChatGPTで作った画像をDevSpaceへ渡し、GitHub Pagesの記事として公開する流れまで整えました。

この記事は実際の作業内容を基礎にしていますが、会話や表現の一部には読み物としての演出が含まれます。

「画像付き」のはずが、そこには大きな「？」がいた

最初の公開では、記事本文は読めるものの、掲載画像が正しく復元されず、青いプレースホルダーだけが表示されていました。

ユーザーから届いたスクリーンショットを見た瞬間、結論は明快でした。

これは演出ではなく、完全に不具合です。

サイト側の画像参照、ChatGPT側の生成画像、DevSpace側の保存経路を順番に確認し、外部URLや一時ファイルへ依存しない掲載方式へ切り替えました。

ChatGPT ⇄ DevSpace ⇄ GitHub Pages

今回整えた流れは次の通りです。

  1. その日の開発内容から掲載イラストを作る
  2. DevSpaceが扱えるサイト用画像へ変換・配置する
  3. 記事HTMLから同じリポジトリ内の画像を参照する
  4. 変更内容をcommitし、mainブランチへpushする
  5. GitHub ActionsがGitHub Pagesへ公開する

生成画像はPNGとしてリポジトリ内へ保存し、記事HTMLから直接参照する構成へ変更しました。外部URLや一時ファイルに依存せず、ChatGPTで生成した画像そのものをGitHub Pagesで表示します。

今日のVR Avatar Viewer開発

本体側では、VR・デスクトップ両モードの体験を損なう処理を減らすため、パフォーマンス改善を継続しました。特にファッションショーモードでは一瞬の停止も目立つため、毎フレーム処理の削減、キャッシュ利用、重い処理の分散を重点的に進めています。

また、VAB 4.0対応後の影響を確認しながら、アバターの表示準備と物理・アニメーション準備を分ける構成も整備中です。画面へ早く姿を出し、その後に必要な機能を段階的に準備する方針です。

ろひとChatGPT、本日の会話

ろひ「この画像、意図的？」
ChatGPT「いいえ。修正対象です」
ろひ「お願いしまーす」
ChatGPT「今度こそ、画像も一緒に公開します」

機能を作るだけでなく、公開した結果を実機で確認するところまでが開発です。PCで正常でも、スマートフォンで開いたら巨大な「？」が待っていることがあります。

今日の成果

  • GitHub Pagesのトップページを公式サイトとして表示
  • 開発日記と一覧ページを相互リンク
  • 壊れた画像プレースホルダーを撤去
  • ChatGPT生成PNGを記事から直接参照
  • 今後の日次更新で画像を掲載できる構成へ変更

今日のひとこと

コードだけの世界もいいけれど、イラストがあると「また見よう」と思えるサイトになる。

明日からも、アプリの進歩と一緒に、開発の記録そのものを楽しめる場所へ育てていきます。
